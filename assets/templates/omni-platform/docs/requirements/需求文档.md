## 1. 背景与目标

### 1.1 业务背景

- 现状：`omni-platform` 已有平台型初始化基础，但文档体系仍偏占位，未完全接入正式交付模板。
- 问题：当前 docs 与后续标准交付链路衔接不够强，需求、设计、测试、发布文档没有按统一模板完整落盘。
- 机会：借用 `product-delivery-skill` 的标准模板，可以把 `omni-platform` 升级成真正可交付、可派生、可验收的平台母版。

### 1.2 核心目标

- 目标 1：让 `docs/` 具备完整的需求、设计、控制、测试、发布模板骨架。
- 目标 2：让后续从 `omni-platform` 派生的新项目，默认就有标准交付证据链入口。
- 目标 3：让 `omni-platform` 成为“平台型项目初始化 + 交付模板”双重母版。

### 1.3 量化指标

| 指标 | 当前值 | 目标值 | 统计口径 |
|---|---|---|---|
| 标准文档模板覆盖数 | 6 左右 | 20+ | docs 内落盘的 product-delivery 标准产物数量 |
| README 门禁通过 | 已通过 | 持续通过 | readme-gate 结果 |
| 三端可构建 | 已通过 | 持续通过 | `pnpm build` |

## 2. 范围定义

- 本轮范围：将 `product-delivery-skill` 的需求、设计、控制、测试、发布模板接入 `omni-platform/docs`，并补索引与关键元信息。
- 非本轮范围：不编写真实业务需求，不补完整业务接口和数据库设计。
- 依赖系统：`product-delivery-skill` 文档模板、共享知识库 README 规范。
- 风险前置假设：模板允许先作为母版骨架存在，后续真实项目再填业务内容。
- 需要协同的角色：产品、架构、前端、后端、测试。

## 3. 用户与场景

| 角色 | 场景 | 关键动作 | 成功标准 |
|---|---|---|---|
| 架构负责人 | 维护平台母版 | 检查结构、部署、模板入口 | 新项目可直接派生 |
| 产品负责人 | 编写新项目需求 | 复用 docs/requirements 模板 | 不再从零建文档骨架 |
| 开发负责人 | 派生新项目 | 复制三端结构与文档模板 | 工程与文档同时起步 |

## 4. 功能需求

| 功能点 | 输入 | 处理规则 | 输出 | 异常处理 | 优先级 |
|---|---|---|---|---|---|
| 文档模板接入 | `product-delivery-skill` 模板文件 | 按模块映射复制到 `docs/` | 标准文档骨架 | 模板缺失则保留原说明并手工补齐 | P0 |
| 文档索引生成 | 已存在 docs 子目录 | 生成统一 `docs/INDEX.md` | 快速导航入口 | 缺文件则保留空链接占位 | P0 |
| 关键元信息回填 | 项目名、日期、状态 | 在核心模板中替换为 `omni-platform` 上下文 | 可读的母版文档 | 未确认字段保留最小可编辑结构 | P1 |

## 5. 数据与字段规则

| 字段 | 来源类型 | 展示格式 | 是否可编辑 | 校验规则 | 备注 |
|---|---|---|---|---|---|
| `request-key` | 系统约定 | kebab-case | 是 | 与项目/需求唯一对应 | 建议与阶段状态一致 |
| `状态` | 系统带出 | 枚举 | 是 | 使用模板约定状态值 | 如 ready / todo / in_progress |
| `证据ID` | 人工填写 | 文本 | 是 | 必须可追溯到命令、截图或日志 | 验收文档强依赖 |

> 来源类型建议：`用户输入 / 系统带出 / 关联回填 / 系统计算 / 隐藏字段`。

### 5.1 字段描述补充要求

- 字段描述必须写清来源、展示格式、是否必填、是否可编辑、回写时机。
- 涉及人员、组织、岗位、部门、编码类字段时，优先说明来源能力而不是只写展示文案。
- 只读字段要区分“前端只读展示”和“后端补齐不可编辑”。

## 7. 流程与状态流转

| 当前状态 | 动作 | 下一状态 | 触发条件 | 非法流转处理 |
|---|---|---|---|---|
| requirement | 完成需求文档 | design | 需求范围与目标明确 | 返回需求补齐 |
| design | 完成技术/设计方案 | development | 结构、实现与验收方式明确 | 返回设计补齐 |
| development | 完成实现与自测 | review | 构建通过且开发放行报告可填写 | 返回开发自测 |

### 7.1 联动与状态补充

- 涉及联动时，必须写清：来源模块 → 目标模块 → 触发时机 → 写入字段 → 状态变化。
- 涉及审批流、撤回、提交、作废、删除时，必须写清前置条件和失败提示。
- 跨模块回写必须说明是否同步刷新、是否需要二次确认、是否可回滚。

## 8. 验收标准

### 8.1 功能验收

- docs 内存在标准模板并能覆盖需求、设计、控制、测试、发布链路。

### 8.2 体验验收

- 首次接手者可以只看 `README.md + docs/INDEX.md` 快速理解文档入口。

### 8.3 稳定性验收

- 文档接入不影响 `README gate`、`pnpm build`、`docker compose config -q`。

### 8.4 证据要求

- 验收结论必须对应证据 ID、截图、日志或测试记录。
- 没有证据链的“通过”不算通过。

## 9. 风险与回滚策略

| 风险 | 影响范围 | 预防措施 | 回滚动作 |
|---|---|---|---|
| 模板过多导致阅读成本上升 | docs 可读性 | 增加 `docs/INDEX.md` 聚合导航 | 保留索引，按需删减低价值模板 |
| 模板元信息不贴合母版 | 后续误用 | 关键模板先回填 omni-platform 上下文 | 再次统一批量修订 |
