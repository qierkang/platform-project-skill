## 1. 架构设计

- 技术目标：将 `product-delivery-skill` 的标准文档模板映射到 `omni-platform/docs`，形成正式母版交付链路。
- 技术边界：本轮只做文档母版接入与目录收口，不新增真实业务功能。
- 依赖服务：`product-delivery-skill`、共享 README 规则、当前 `omni-platform` 三端骨架。
- 模块拆分：`requirements / design / architecture / control / testing / release / api / config / documents / deployment`。

## 2. 数据设计

### 2.1 表结构变更

| 表名 | 动作（新增/变更） | 说明 | 回滚策略 |
|---|---|---|---|
| 文档目录结构 | 变更 | 新增 control/release/config 并导入模板 | 删除新增目录并回退 docs 索引 |

### 2.1.1 SQL 可读性规范（强制）

1. 建表 SQL 必须包含中文表注释与字段注释。
2. 索引必须注明用途（查询、统计、排序等）。
3. SQL 文件头必须写作者：`xyqierkang@gmail（claude+codex）`。

### 2.2 字段规则

| 字段 | 前端传值 | 后端补齐 | 校验规则 | 默认值 |
|---|---|---|---|---|
| `request_key` | 无 | 文档维护时填写 | 与需求总览保持一致 | `omni-platform-formal-template` |
| `current_stage` | 无 | 文档维护时填写 | 必须与 stage-status 对齐 | `design` |

## 3. 接口设计

| 接口 | 方法 | 入参 | 出参 | 鉴权 | 错误码 |
|---|---|---|---|---|---|
| `/health` | GET | 无 | 服务状态 | 否 | 500 |
| `/meta` | GET | 无 | 项目元信息与端口信息 | 否 | 500 |

## 4. 关键流程时序

1. 从 `product-delivery-skill/shared/templates/` 选择模板并映射到 `omni-platform/docs/`。
2. 对需求总览、需求文档、技术方案、阶段状态等关键文件回填母版上下文。
3. 通过 `docs/INDEX.md` 聚合导航，保证首次接手可快速定位。

## 5. 非功能设计

| 维度 | 要求 | 实现策略 |
|---|---|---|
| 性能 | 文档入口可快速定位 | 聚合索引 + 分层目录 |
| 安全 | 不破坏现有构建与部署入口 | 仅新增 docs 模板，不改运行逻辑 |
| 可观测性 | 文档状态可追踪 | `stage-status.json` + 需求总览 |
| 可回滚性 | 模板接入可撤销 | 新增目录可独立删除或回退 |
